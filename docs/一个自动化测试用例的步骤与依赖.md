# 一个自动化测试用例的步骤与依赖

下面用项目里真实的一个用例，说明「从写到跑」的完整流程和依赖关系。

---

## 一、一个用例具体长什么样

以 **ContentDetector** 的其中一个用例为例：

**文件位置**：`src/converters/ContentDetector.test.ts`

```ts
import { describe, it, expect } from 'vitest';
import { ContentDetector } from './ContentDetector';
import { ContentType } from '../types';

describe('ContentDetector', () => {
  it('应识别 10 位秒级时间戳', () => {
    const r = ContentDetector.detect('1704067200');
    expect(r.type).toBe(ContentType.TIMESTAMP_SECOND);
    expect(r.value).toBe('1704067200');
    expect(r.confidence).toBeGreaterThanOrEqual(0.9);
  });
});
```

含义简述：

| 部分 | 作用 |
|------|------|
| `describe('ContentDetector', () => { ... })` | 把一组用例归到一个「套件」里，名字会出现在测试报告里 |
| `it('应识别 10 位秒级时间戳', () => { ... })` | **一条具体用例**：描述期望行为，里面写「怎么测」 |
| `ContentDetector.detect('1704067200')` | 调用被测代码，得到实际结果 |
| `expect(r.type).toBe(...)` | **断言**：实际结果是否和预期一致，不一致则这条用例失败 |

所以：**一条自动化测试用例 = 调用被测代码 + 用 expect 做断言**。

## 二、执行一条用例的完整步骤（在你这台机器上发生了什么）

当你执行 `npm run test:run` 时，大致顺序是：

1. **npm 执行脚本**  
   `package.json` 里 `"test:run": "vitest run"`，所以实际运行的是 `vitest run`。

2. **Vitest 读配置**  
   会加载项目根目录的 `vitest.config.ts`，其中有一行：
   ```ts
   include: ['src/**/*.test.ts', 'src/**/*.test.tsx']
   ```
   所以 Vitest 会**自动发现**所有匹配的测试文件（包括 `ContentDetector.test.ts`）。

3. **运行测试文件**  
   - 用 Node 环境（`environment: 'node'`）执行这些文件。  
   - 遇到 `describe` / `it` 就登记用例，遇到 `expect(...)` 就做断言。  
   - 若某个 `expect` 不满足（例如 `r.type` 实际不是 `TIMESTAMP_SECOND`），Vitest 会标记这条用例失败并打印期望 vs 实际。

4. **输出结果**  
   终端里会看到每个文件的通过/失败、以及每条 `it` 的名字（如「应识别 10 位秒级时间戳」）。

所以：**一条用例的执行路径 = npm 脚本 → Vitest → 找到 \*.test.ts → 执行 describe/it → 执行 expect → 根据结果报通过/失败**。

## 三、依赖是什么（需要装什么、各自干什么）

### 1. 运行时依赖（必须）

| 依赖 | 作用 |
|------|------|
| **vitest** | 测试运行器：发现 `*.test.ts`、执行 `describe`/`it`、运行 `expect`、输出报告。没有它就没有「跑测试」这一步。 |
| **TypeScript + Vite/tsconfig** | 项目本身就在用 TS；Vitest 会按项目配置编译 TS，所以用例可以写在 `.test.ts` 里并 import 业务代码。 |

也就是说：**能跑起来一个用例，最少需要「Vitest + 项目现有的 TS 环境」**。

### 2. 写用例时用到的 API（来自 Vitest）

- `describe`、`it`：组织用例和单条用例。  
- `expect`：断言（如 `expect(x).toBe(y)`、`expect(arr).toHaveLength(3)`）。  

这些都在 `vitest` 包里，通过：

```ts
import { describe, it, expect } from 'vitest';
```

使用。**不依赖 Jest**，Vitest 自带的。

### 3. 被测代码的依赖（以 ContentDetector 为例）

- `ContentDetector` 来自 `./ContentDetector`，只用了 `../types` 里的类型和枚举。  
- 在 **Node 环境** 下运行，不需要浏览器、不需要 Chrome 扩展 API。  

所以这条用例**只依赖**：  
- 项目里的 `ContentDetector.ts`  
- 项目里的 `types`  
- Vitest 提供的 `describe` / `it` / `expect`  

没有额外 npm 包。

### 4. 可选依赖（当前用例没用到，但项目已装好）

| 依赖 | 用途 |
|------|------|
| **@vitest/coverage-v8** | `npm run test:coverage` 时生成覆盖率（v8 引擎统计）。 |
| **jsdom** / **@testing-library/react** 等 | 以后给 React 组件写测试时用（需要 DOM 环境）。当前这些用例是纯 Node 逻辑，用不到。 |

总结：**执行你现在这些自动化测试用例，真正用到的就是 Vitest + 项目里的业务代码和类型；** 其它包是给「覆盖率」和「以后的组件测试」准备的。

## 四、自己新写一条用例的标准步骤

1. **定被测对象**  
   例如：`src/utils/xxx.ts` 里的 `formatDate` 函数。

2. **在同目录或就近建测试文件**  
   文件名以 `.test.ts` 结尾，且落在 `src/**` 下，例如：  
   `src/utils/xxx.test.ts`。

3. **写「调用 + 断言」**  
   ```ts
   import { describe, it, expect } from 'vitest';
   import { formatDate } from './xxx';

   describe('formatDate', () => {
     it('应格式化为 YYYY-MM-DD', () => {
       expect(formatDate(1704067200000)).toBe('2024-01-01');
     });
   });
   ```

4. **运行**  
   ```bash
   npm run test:run
   ```
   或只跑这个文件：
   ```bash
   npx vitest run src/utils/xxx.test.ts
   ```

5. **看终端**  
   - 通过：该条用例会显示为绿色/通过。  
   - 失败：Vitest 会打印期望值和实际值，方便改代码或改用例。

---

## 五、和「手动功能测试」的区别（一句话）

- **自动化测试用例**：由 **Vitest 在 Node 里执行**，只跑你写的 `describe`/`it`/`expect`，不打开浏览器、不加载扩展；适合回归逻辑、适合 CI。  
- **手动功能测试**：你在浏览器里**真人操作**扩展（点 Popup、选文字、看悬浮窗等），按《功能测试用例》文档一步步勾选；适合验证真实交互和 UI。

两者互补：改完代码先跑 `npm run test:run`，发版前再按文档做一遍手动测试即可。
