# 从零写一个自动化测试用例（Vitest）

面向「没写过 Vitest、想自己从 0 写一条用例并后续维护」的实操指南。

---

## 一、环境准备（项目里已有可跳过）

### 1. 安装 Vitest

```bash
npm install -D vitest
```

### 2. 加运行脚本

在 `package.json` 的 `scripts` 里加上：

```json
"test": "vitest",
"test:run": "vitest run"
```

- `npm run test`：监听模式，改代码自动重跑。
- `npm run test:run`：单次跑完就退出（适合确认「全绿」再提交）。

### 3. 配置文件（可选但推荐）

在项目根目录建 `vitest.config.ts`：

```ts
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    environment: 'node',
    include: ['src/**/*.test.ts', 'src/**/*.test.tsx'],
  },
});
```

这样 Vitest 只会跑 `src` 下以 `.test.ts` / `.test.tsx` 结尾的文件。

---

## 二、从零写第一条用例（4 步）

### 第 1 步：选「要测谁」

选一个**纯函数或类方法**（不依赖浏览器、不依赖 Chrome API），例如：

- `src/converters/ContentDetector.ts` 里的 `ContentDetector.detect(text)`
- 或你自己写的 `src/utils/format.ts` 里的 `formatDate(ts)`

下面用 `ContentDetector.detect` 举例。

### 第 2 步：建测试文件

- **位置**：和被测代码**同目录**或**同模块**，方便以后一起改。
- **命名**：`被测文件名.test.ts`（或 `被测模块名.test.ts`）。

例如被测是 `src/converters/ContentDetector.ts`，就建：

```
src/converters/ContentDetector.test.ts
```

### 第 3 步：写「调用 + 断言」

在 `ContentDetector.test.ts` 里写：

```ts
import { describe, it, expect } from 'vitest';
import { ContentDetector } from './ContentDetector';
import { ContentType } from '../types';

describe('ContentDetector', () => {
  it('识别 10 位秒级时间戳', () => {
    const r = ContentDetector.detect('1704067200');
    expect(r.type).toBe(ContentType.TIMESTAMP_SECOND);
    expect(r.value).toBe('1704067200');
  });
});
```

含义：

| 你写的 | 作用 |
|--------|------|
| `describe('ContentDetector', () => { ... })` | 这一组用例都测 ContentDetector，名字会出现在报告里 |
| `it('识别 10 位...', () => { ... })` | **一条用例**：描述「期望行为」 |
| `ContentDetector.detect('1704067200')` | 调用被测代码 |
| `expect(r.type).toBe(...)` | 断言：实际值必须等于预期，否则用例失败 |

### 第 4 步：运行

```bash
npm run test:run
```

终端里看到这一条通过，就说明「从零写的一条用例」已经跑通了。

## 三、一条用例的「标准写法」模板

以后每次新写用例，可以按这个结构套：

```ts
import { describe, it, expect } from 'vitest';
import { 你要测的函数或类 } from './被测文件';

describe('被测模块或类名', () => {
  it('期望行为的一句话描述', () => {
    const 结果 = 调用被测代码(输入);
    expect(结果.某字段).toBe(期望值);
    // 需要的话多写几行 expect
  });
});
```

- **describe**：按「模块/类/文件」分组，可以嵌套。
- **it**：一条用例只测一种情况；多种情况就写多个 `it`。
- **expect**：只写「和预期不一致就会失败」的断言。

---

## 四、常用断言（expect）写法

| 需求 | 写法 |
|------|------|
| 等于某值 | `expect(x).toBe(1)`、`expect(s).toBe('ok')` |
| 不相等 | `expect(x).not.toBe(2)` |
| 真/假 | `expect(flag).toBe(true)`、`expect(err).toBeFalsy()` |
| 数组长度 | `expect(arr).toHaveLength(3)` |
| 包含/匹配 | `expect(s).toMatch(/^\d{4}-/)`、`expect(arr).toContain(2)` |
| 抛错 | `expect(() => fn()).toThrow('错误信息')` |
| 数值比较 | `expect(n).toBeGreaterThanOrEqual(0.9)` |

更多见 [Vitest 文档 Expect](https://vitest.dev/api/expect.html)。

## 五、后续如何维护

### 1. 加新用例

- 在**同一个** `describe` 里加新的 `it(...)`，或新开一个 `describe`。
- 保持「一个 it 只测一种情况」，名字写清楚期望行为。

```ts
it('识别 13 位毫秒时间戳', () => {
  const r = ContentDetector.detect('1704067200000');
  expect(r.type).toBe(ContentType.TIMESTAMP_MILLISECOND);
});
```

### 2. 业务代码改了怎么办

- **行为没变**：一般不用改测试，跑一遍 `npm run test:run` 确认仍通过即可。
- **行为变了**（例如返回值、入参、错误类型变了）：
  - 先跑测试，看哪几条报错。
  - 根据新逻辑改对应用例里的**输入**或 **expect 的期望值**，直到全绿。
  - 若新行为需要新场景，就新写一个 `it`。

原则：**测试要跟着「约定好的行为」走**；行为变更时，测试一起改。

### 3. 什么时候跑测试

- 改完和「被测逻辑」相关的代码后：本地跑一次 `npm run test:run`。
- 提交前：建议养成习惯跑一遍，避免把断掉的用例提交上去。
- 可选：用 `npm run test` 开监听，改代码时自动重跑，方便边改边看。

### 4. 只跑某一个文件

```bash
npx vitest run src/converters/ContentDetector.test.ts
```

方便你只验证刚改的那一组用例。

### 5. 维护时的注意点

- **用例名**：`it('...')` 里用一句话说清「在什么情况下、期望什么」，以后看报告能直接看懂。
- **不要测实现细节**：只断言「对外行为」（返回值、抛错、副作用结果），不要断言内部变量或私有函数。
- **一个 it 一个场景**：失败时能立刻知道是哪种情况挂了；避免一个 it 里测很多种输入。

## 六、小结

| 步骤 | 你要做的 |
|------|-----------|
| 从零写一条用例 | 1）选一个函数/方法 2）同目录建 `xxx.test.ts` 3）`describe` + `it` + 调用 + `expect` 4）`npm run test:run` |
| 维护 | 加场景就加 `it`；业务行为变了就改对应用例的输入或期望；提交前跑一遍测试 |
| 依赖 | 写用例只需要：`vitest`（describe/it/expect）+ 你的业务代码；跑起来用 `npm run test` 或 `npm run test:run` |

按上面做，你就可以在项目里从 0 自己写、并长期维护 Vitest 自动化测试用例。
